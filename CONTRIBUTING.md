# Contributing to Prax

Thanks for helping build Prax. This project is optimized for practical, reusable output for solo builders.

## Contribution Principles

- Action over theory: every contribution should improve execution quality.
- Low friction: docs and steps should remain easy for non-technical users.
- Reusable by default: templates/modules should be fork-friendly.
- Explain the why: avoid black-box instructions when writing docs.

## Where to Contribute First

1. `modules/*` learning module clarity and runability
2. `templates/module-template.md` improvements
3. onboarding docs (`README.md`, `docs/*`)
4. issue and PR template improvements

## Development Flow

1. Fork repository
2. Create branch (`feat/<topic>` or `fix/<topic>`)
3. Make focused changes
4. Run a basic self-check:
   - steps are runnable
   - file paths are valid
   - docs are consistent
5. Open a Pull Request

## Module Quality Checklist

For any new module, include:

- Clear context and target user
- Time-boxed goal (prefer <= 30 minutes to first useful output)
- Copy-paste runnable steps
- Expected output example
- Common failure cases and fixes
- One extension task for advanced users

## Commit Message Convention

Use imperative and scope:

- `feat(module): add ai-digest module skeleton`
- `docs(readme): clarify quick start`
- `fix(template): correct path in module template`

## Pull Request Expectations

- Explain what changed and why
- Link related issue if available
- Include before/after behavior
- Keep PR small and reviewable

## Code of Conduct

By participating, you agree to follow `CODE_OF_CONDUCT.md`.
