# Prax Roadmap

> **版本**: v2 (2026-04-08)
> **路线图形态**: 企业级 AI Agent 场景模板库
> **主战场平台**: Dify（开源、自托管、workflow 可导出 YAML）
> **兼容战场**: Coze Studio 开源版、n8n

## 2. Phase 1 — Deploy（0-8 周）

### 2.1 目标

验证核心假设：**有产品思维但不写代码的从业者，能在 30 分钟内通过 Prax 模板在 Dify 上部署一个真正服务业务的 Agent**。

### 2.2 交付范围

1. 建立企业级 Agent 模板标准：`workflow.yaml + configs + docs` 三件套
2. 发布 3 个首发场景模板（见 2.3）
3. 每个模板提供：
   - Dify workflow 配置文件（可直接导入）
   - 示例数据源与输出样本
   - 一键部署指南（含 API Key 准备、渠道配置）
   - 5 步内可跑通的 Quick Start
4. 建立模板质量门槛文档（Template Quality Checklist）

### 2.3 首发模板（企业场景化）

- **`ai-digest`**：RSS 多源聚合 → AI 筛选摘要 → 邮件每日推送
  - 业务场景：行业情报岗、产品经理每日资讯
  - MVP 范围：3-5 个 RSS 源 + AI 摘要 + 邮件单渠道
- **`content-repurpose`**：单篇稿件 → 多平台格式自动改写（小红书/公众号/视频脚本）
  - 业务场景：内容运营、独立创作者
  - MVP 范围：1 份输入 → 3 份平台适配输出
- **`workflow-starter`**：最小可复用 Agent 编排模板（"输入-处理-输出"骨架）
  - 业务场景：通用可 fork 起点
  - MVP 范围：标准 workflow 模板 + fork 指南

### 2.4 验证指标

- `DT (Deploy Time)` 中位数 <= 30 分钟
- `FDR (First Deploy Rate)` >= 60%
- 每模板至少 3 位企业从业者成功部署
- 模板运行稳定性 >= 80%（10 次运行无阻塞错误）

### 2.5 主要风险与缓解

- **风险**：用户卡在"Dify 首次部署/注册"步骤  
  **缓解**：提供 Dify Cloud 与本地 Docker 两种路径，均写入模板 README
- **风险**：API Key 配置对非技术用户是门槛  
  **缓解**：每个模板 README 第一节就是"API Key 准备清单"，包含申请链接 + 成本估算
- **风险**：RSS 源不稳定，Agent 跑挂  
  **缓解**：在 workflow 中加 HTTP Retry 与 Fallback 节点
- **风险**：模板与真实业务脱节  
  **缓解**：每个模板上线前邀请 2 位目标用户做可用性测试

## 3. Phase 2 — Customize（9-16 周）

### 3.1 目标

让单场景模板变成"**多配置可适配**"，覆盖企业真实复杂度。

### 3.2 交付范围

1. 多源支持：RSS + 邮件订阅 + 网页定时抓取 + Webhook
2. 多推送渠道：邮件 + 飞书 + 企业微信 + Slack + GitHub Release
3. 场景预设：为不同岗位（PM / 运营 / BA / 开发者）提供预配置包
4. 配置标准化：统一 `configs/*.yaml` 规范，降低 fork 定制成本
5. 观测与日志：提供模板级日志与错误追踪最佳实践

### 3.3 验证指标

- 每模板支持至少 3 种数据源、3 种推送渠道
- 二次定制（改配置不改结构）用户比例 >= 40%
- 至少 10 家企业/团队部署过其中一个模板

### 3.4 主要风险与缓解

- **风险**：配置标准化过度 vs 灵活性不足  
  **缓解**：区分"核心配置"（必填、稳定）与"高级配置"（可选、可扩展）
- **风险**：多渠道集成维护成本高  
  **缓解**：建立社区贡献机制，每个渠道由一位 Maintainer 负责

---

## 4. Phase 3 — Share（17-24 周）

### 4.1 目标

让 Prax 从"官方模板库"演变为"**社区模板市场 + 企业落地案例集**"。

### 4.2 交付范围

1. 模板市场：可检索、可筛选、可分级（官方 / 推荐 / 实验）
2. 企业落地案例集：每个案例包含业务背景、使用模板、定制方式、效果数据
3. 贡献者流程：从提交 Issue 到合并模板的完整 SOP（Standard Operating Procedure）
4. 模板质量分级体系：官方认证、社区推荐、实验阶段三档

### 4.3 验证指标

- 社区贡献模板 >= 10 个
- 企业公开案例集 >= 20 个
- 单模板平均 fork 数 >= 3
- 月度社区活跃贡献者 >= 5 人

### 4.4 主要风险与缓解

- **风险**：冷启动导致社区内容稀缺  
  **缓解**：先由核心团队发布 10+ 模板，邀请种子贡献者
- **风险**：质量参差影响信任  
  **缓解**：严格模板质量门槛 + 三档分级 + 用户评价机制

## 5. 横向工作流（跨阶段持续）

### 5.1 文档体系

| 文档 | 用途 |
|---|---|
| `README.md` | 对外叙事、一句话定位、快速入口 |
| `docs/vision.md` | 战略方向与边界 |
| `docs/roadmap.md` | 执行计划与里程碑 |
| `docs/testing/first-run-checklist.md` | 用户测试脚本（Phase 1 完成前建立） |
| `docs/governance/module-quality-checklist.md` | 模板质量门槛（Phase 1 完成前建立） |
| `CONTRIBUTING.md` | 贡献流程规范（Phase 2 完善） |

### 5.2 数据观测

建立最小观测面板，追踪：

- 每模板首次部署成功率、卡点步骤
- 用户实际运行次数（模板被真实使用的频率）
- 社区 fork、star、PR（Pull Request）行为

### 5.3 品牌内容联动

每个模板至少对应一条公开内容资产：

- 模板发布贴（你做了什么模板、对应什么企业场景）
- 复盘贴（为什么这么设计、踩过什么坑）
- 企业案例贴（真实业务应用故事）

## 6. 发布节奏建议（双周迭代）

采用双周 Sprint：

- 第 1 周：模板设计 + workflow 实现 + 文档草稿
- 第 2 周：可用性验证 + 修复 + 发布 + 复盘

每个 Sprint 固定输出：

1. 一个可运行的新模板或增量
2. 一篇公开说明（发布贴或复盘贴）
3. 一份数据复盘（DT / FDR / 用户反馈）
